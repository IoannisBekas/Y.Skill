# Project Health Pass

Use these instructions only for a requested deep project-wide health pass, repo reliability sweep, audit-only report, fix-verified remediation pass, or baseline-diff comparison.

Preserve user changes. Select exactly one mode before scanning: `audit-only`, `fix-verified`, or `baseline-diff`. First read `.agent_memory.json` if present and treat it as advisory memory for verified false positives, command quirks, effective strategies, failed assumptions, and project idiosyncrasies. Map the repository before editing: layout, git status, instructions, manifests, lockfiles, CI, scripts, docs, and source/generated boundaries. Use existing package managers and scripts. Run native checks where safe. Maintain an evidence coverage matrix: `pass`, `finding`, `unknown`, or `not-applicable`; unknown is not healthy. Rank findings as P0 security/data loss/destructive/OOM, P1 runtime/build/deploy blocker, P2 silent failure/reliability/major validation or typing gap, P3 cleanup/debt. Assign stable finding IDs like `HP-SEC-001`, `HP-BUILD-001`, and `HP-TEST-001`; reuse IDs across reruns for the same root cause. Counter-review P0/P1/P2 candidates before reporting or fixing. Build a safest-first remediation backlog with blast radius, reversibility, verification, rollback, and status. Fix verified P0/P1/P2 issues only in `fix-verified` mode and only when safely scoped. If a patch breaks checks, critique the wrong assumption before trying a second fix. Add focused tests. In `audit-only`, do not write memory or source files. If requested, emit and validate `project-health-pass.report.v1` JSON. Before final response, append verified non-sensitive lessons to `.agent_memory.json` unless mode is `audit-only`. Report mode, scanned components, coverage, severity-grouped `HP-*` problems, modified files, checks run, memory status, machine report status, suppressions/accepted risks, safest first batch, and blocked next steps.
